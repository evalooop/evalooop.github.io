Understanding the ASL Metric and EVALOOP Evaluation Framework

EVALOOP Evaluation Framework

EVALOOP exploits the natural duality between code generation and code summarization tasks to create a self-contained feedback loop for robustness evaluation.

EVALOOP Process

Initial Code Generation

Given a natural language specification, the LLM generates code solution

Code Summarization

The generated code is summarized by the same LLM into a new specification

Loop Iteration

Use the new specification to generate code again, creating a feedback loop

Functional Testing

Test generated code against original test cases to verify correctness

Loop Continuation

Continue the loop until generated code fails functional tests

ASL Calculation

Compute the average number of sustainable loops before failure

Evaluation Benchmark

MBPP+ (Mostly Basic Python Problems Plus)

EVALOOP uses the MBPP+ benchmark, an enhanced version that applies rigorous quality filtering and expands the test case corpus by 35-fold compared to the original MBPP dataset.

Dataset Statistics:
  • 378 programming tasks
  • Python function synthesis
  • Natural language descriptions
  • 35x more test assertions vs. original MBPP
  • Rigorous quality filtering applied
Evaluation Setup:
  • Maximum 10 loops per task
  • Pass@1 metric at each iteration
  • Quadratic weighting (i²) for ASL
  • GPT-4-turbo for similarity evaluation
  • Greedy decoding (temperature=0.0)

Average Sustainable Loops (ASL) Metric

ASL (Average Sustainable Loops) measures LLM robustness in programming through self-consistency evaluation using iterative code generation and summarization loops.

Key Innovation

EVALOOP exploits the natural duality between complementary software engineering tasks (code generation and code summarization) within a self-contained feedback loop, providing a unified metric for robustness without relying on external adversarial attacks.

Definition

The ASL metric quantifies model robustness by measuring the average number of sustainable loops before failure in the code generation-summarization cycle:

\[ ASL = \frac{\sum_{i=1}^{M} (n_i \times i^2 \times s_i)}{M \times T} \]
where:
  • M = maximum number of evaluation loops (e.g., 10)
  • ni = number of tasks that sustain functionality through exactly i loops
  • T = total number of tasks in benchmark dataset
  • si = average semantic similarity score for tasks that sustain functionality through exactly i loops
  • i2 = quadratic weighting emphasizing sustained performance

Key Components

Code Generation

LLM generates code from natural language specification

Code Summarization

LLM summarizes code back to natural language

Loop Iteration

Maximum 10 sustainable transformation loops

Semantic Consistency

LLM-based similarity evaluation at failure boundaries

Semantic Similarity Evaluation

A critical component of ASL is the semantic similarity evaluation at failure boundaries. For successful loops, similarity is assumed to be 1.0 since functionally correct programs should be semantically equivalent. For failure boundaries, we use LLM-based evaluation:

\[ s_j = \begin{cases} 1.0 & \text{for successful loops } (j < l) \\ \text{Sim}(\text{prompt}_l, \text{prompt}_{l+1}) & \text{for failure boundary } (l < M) \end{cases} \]

LLM-based evaluator prompt:

Compare the semantic similarity of these two code generation prompts. Consider both the prompts themselves and their generated code outputs.

Prompt 1: {prompt1}
Generated Code 1: {code1}

Prompt 2: {prompt2}  
Generated Code 2: {code2}

Return only a similarity score between 0 and 1, where:
- 1.0 = semantically equivalent (same intent, requirements, and expected output)
- 0.0 = completely different semantic meaning
- Values between 0 and 1 represent partial semantic overlap
